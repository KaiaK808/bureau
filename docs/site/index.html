Quickstart

Six steps from clean repo to autonomous pipeline. No backend, no DB — state lives in your repo and in Linear.

# 1. Install the skill
mkdir -p ~/.claude/skills
git clone https://github.com/KaiaK808/bureau.git ~/.claude/skills/bureau-init

# 2. In Claude Code, authenticate Linear MCP
/mcp     # then choose linear-server

# 3. Add your Linear API key to the target repo
echo 'LINEAR_API_KEY=lin_api_xxxxxxxxxxxx' >> .env

# 4. Bootstrap
/bureau-init

# 5. Launch the pipeline
./scripts/start-bureau-v2.sh

# 6. Watch it work
tmux attach -t bureau-v2-$(basename "$PWD")
Prereqs Claude Code · git · jq · tmux · curl · a Linear account · GitHub CLI gh (recommended). macOS: brew install jq tmux gh.

Pipeline architecture

Each Linear issue moves through a state machine. One agent per state. Each agent is a tmux pane running queue-loop-supervised.sh, polling its state at the configured interval.

Triage ─▶ Spec ─▶ Spec Review ┬─▶ Design ─▶ Build ─▶ QA ─▶ Build Review ┬─▶ Merge ─▶ Done
                              │                                          │
                              └─▶ Copy ──────────────────▶ Build         └─▶ (auto-review loop)

Labels:  lane-2 (eligibility) · needs-ux · needs-copy · needs-human (parking)

House rules

Hard-learned. Each carries an EXP-### incident tag in SKILL.md Appendix B.

  • Never spawn $CLAUDE for Linear CRUD. Linear glue goes through bureau-config.sh helpers — $CLAUDE is reserved for creative work.
  • Branches are resolved via the <!-- bureau-branch: ... --> marker comment, not Linear's branchName.
  • Preconditions run before state mutations. Every pipeline calls precondition_linear (exit 10) and precondition_claude_auth (exit 16) before any move_issue.
  • Exit codes are a protocol — see FILE #02.
  • Worktrees reset between picks. queue-loop.sh fetches, hard-resets, and clean -fdx before each pick.
  • pick_issue filters by label name, not UUID. Custom labels work even when only the main label's UUID was captured.
