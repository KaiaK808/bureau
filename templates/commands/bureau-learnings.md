---
description: >
  Reads logs/events.jsonl and the corresponding Linear comments, then drafts a
  human-curated LESSONS.md surfacing recurring failure modes, repeating review
  feedback, and stage timing patterns from the last 30 days. Never commits.
---

## Purpose

The bureau pipeline emits a structured event per stage run to `logs/events.jsonl`
(see `emit_event` in `scripts/bureau-config.sh`). This skill mines that log plus
the Linear comments it references, clusters the signal, and writes a **draft**
`LESSONS.md` at the repo root. The human reviews, edits, and commits.

Pipelines selectively read `LESSONS.md` back in as advisory context (spec +
code-review only). That feedback loop is the whole point — but it only works
when a human gates the loop, so this skill **must never commit**.

## When to run

- After ~1 week of pipeline activity, the first time.
- Then weekly (or after any noticeable run of failures) to refresh the draft.
- Always before bumping a model override or tightening a pipeline prompt — the
  draft is the cheapest source of evidence for whether the change is needed.

## Workflow

### 1. Preflight

Run from the repo root. Verify:
- `logs/events.jsonl` exists and is non-empty.
- `.bureau.json` exists (needed for team/label context).
- `jq` is available.

If `events.jsonl` is missing or empty, write a one-line placeholder LESSONS.md:

```markdown
# Lessons learned

_No pipeline events yet. Re-run `/bureau-learnings` after the queue has processed some issues._
```

…and stop. Print: `No events yet — placeholder written.`

### 2. Window the data

Compute the cutoff timestamp: 30 days ago in ISO-8601 UTC.

```bash
CUTOFF=$(date -u -v-30d '+%Y-%m-%dT%H:%M:%SZ' 2>/dev/null || date -u -d '30 days ago' '+%Y-%m-%dT%H:%M:%SZ')
```

Filter events.jsonl to the window:

```bash
jq -c --arg cutoff "$CUTOFF" 'select(.ts >= $cutoff)' logs/events.jsonl > /tmp/events-window.jsonl
```

If the windowed file is empty, treat it as the "no events" case from step 1.

### 3. Identify failed issues (for failure-mode clustering)

A failure for our purposes is any `stage_end` event where `class` is not in
`["ok", "queue-empty"]`. Group by `issue`:

```bash
jq -sc 'group_by(.issue) | map({
  issue: .[0].issue,
  failures: [.[] | select(.event=="stage_end" and (.class | IN("ok","queue-empty") | not))]
}) | map(select(.failures | length > 0))' /tmp/events-window.jsonl > /tmp/failed-issues.json
```

For each failed issue:
1. Call `mcp__linear-server__list_comments` for the issue. Pull the most recent
   5 comments.
2. From the comments, extract the **last error context** — pipelines post a
   crash trail via their EXIT traps. Look for:
   - Stack traces (lines starting with `at `, `File "...", line`, `Traceback`).
   - The first file path mentioned in such a trail.
   - Any `**Build**: FAILED` block from code-review-pipeline.
3. Bucket the failure by `(class, first_file_in_trace)`. If no file is
   identifiable, bucket by `(class, last_pipeline_stage_run)`.

Keep only buckets with ≥3 distinct issues. **Do not fabricate clusters** with
fewer — say "below threshold" in the section preamble and move on.

### 4. Identify repeating review feedback

A successful-but-reviewed issue has a `stage_end` for
`code-review-pipeline.sh` with `exit_code=0` in the window. For each such
issue:
1. Call `mcp__linear-server__list_comments`.
2. Find the latest comment whose body contains `## Code Review v2 — `.
3. Extract the body of that comment (everything after the marker line through
   the next horizontal rule or end-of-comment).

Across all extracted review bodies:
- Tokenize into sentences.
- Find n-grams (4–8 words) that appear in **≥3 distinct issues** within the
  window. Lowercase, drop punctuation for matching but quote the original
  spelling in the output.
- Group similar n-grams (lexical overlap ≥70%) under one bullet.

Skip the section if no qualifying n-grams. Don't reach for filler.

### 5. Stage timing

```bash
jq -sc '
  [.[] | select(.event=="stage_end" and .duration_s != null)]
  | group_by(.stage)
  | map({
      stage: .[0].stage,
      n: length,
      p50: (map(.duration_s) | sort | .[length/2|floor]),
      p90: (map(.duration_s) | sort | .[(length*0.9)|floor])
    })
' /tmp/events-window.jsonl > /tmp/stage-timing.json
```

Include all stages with `n ≥ 5` runs. Skip the section if no stage clears the
bar.

### 6. Write the draft

Overwrite `LESSONS.md` at the repo root. Use this structure:

```markdown
# Lessons learned

_Draft generated by `/bureau-learnings` from logs/events.jsonl + Linear comments.
Window: last 30 days. Threshold: ≥3 distinct issues per finding. Edit freely
before committing — this is advisory context for future spec and code-review
runs, not a binding rule._

_Generated: <ISO-8601 UTC>_

## Recurring failure modes

<one bullet per cluster, with issue IDs in parens, e.g.:>
- **build-failed in `src/pipeline/foo.rs`** — 4 issues hit `cargo build` errors
  in this file. (EXP-512, EXP-518, EXP-531, EXP-540) Last error preview:
  `error[E0277]: the trait bound ... is not satisfied`.

<or, if nothing clusters:>
- _No failure modes clear the 3-issue threshold this window._

## Review feedback that repeats

<one bullet per n-gram cluster, with quoted phrase + issue list, e.g.:>
- _"the spec doesn't define what happens when X is null"_ — appeared in 5
  review comments (EXP-489, EXP-501, EXP-507, EXP-512, EXP-522). Likely a
  spec-pipeline prompt fix: prompt should require an "edge cases" section.

## Stage timing (p50 / p90, seconds)

| Stage | Runs | p50 | p90 |
|---|---:|---:|---:|
| spec-pipeline.sh | 18 | 142 | 287 |
| ... |

<or skip entirely if no stage has ≥5 runs>

---

_To apply: edit anything that's wrong, commit `LESSONS.md`. The spec and
code-review pipelines will read it on the next run. To dismiss a finding,
delete its bullet — the next `/bureau-learnings` run will re-propose it if
the pattern persists._
```

### 7. Report

After writing, print exactly:

```
LESSONS.md draft written.
- N failure-mode clusters
- N review-feedback clusters
- N stages timed
Review the diff, edit as needed, then `git add LESSONS.md && git commit`.
```

**Do not stage, do not commit, do not push.** If LESSONS.md already exists,
overwrite it — the human's curated version is in git history and they'll merge
diffs by hand.

## Guardrails

- **30-day window, ≥3-issue threshold.** Do not lower these thresholds to
  generate more findings. Empty sections are a valid output — they mean the
  data doesn't support a claim yet.
- **No fabrication.** Every cluster must cite ≥3 issue IDs from the actual
  event log. If you can't, leave the section's bullet list empty with the
  "below threshold" note.
- **Never commit.** This is the entire point — the human gates the feedback
  loop. The skill writes to the working tree only.
- **No network beyond Linear MCP.** No external LLM calls, no embeddings, no
  vector DBs. `jq` + Linear comments is the whole toolkit.

## Error handling

- `logs/events.jsonl` missing/empty → placeholder LESSONS.md, exit clean.
- `jq` not installed → report and stop. Suggest `brew install jq`.
- Linear MCP unavailable → continue with what events.jsonl alone gives you
  (skip the comment-extraction steps; failure-mode bullets list issue IDs only).
- A specific issue's comments fail to fetch → skip that issue, keep going.
